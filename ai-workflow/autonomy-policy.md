# Autonomy Policy — 自治與檢查點政策

> 狀態:**v0.1 draft** — 初版,待真功能試跑後修正。
> 定義 AI 在工作流中**哪裡可以自動衝、哪裡必須停下找人**。

---

## 0. 核心心法

> 監督式自治,不是 fire-and-forget。

本工作流的目標不是「AI 一次把所有程式寫完」(原始文件 §17),而是「在清楚閉環中持續推進」。
所以「完全放行 = 自己跑完全部」是**錯誤期待**,而且會跟工作流自己的停下規則打架。

正確玩法:**讓 AI 在「blast radius 小」的地方高度自動,在「會擴散的決策點」設人類檢查點。**
早期一個錯的架構 / 資料決策,在全自動長鏈裡會在錯地基上蓋很多東西,等 review 才發現就晚了。

---

## 1. 自治分級(依 blast radius)

| 級別 | 範圍 | 自治程度 |
|---|---|---|
| **L3 全自動** | Fast-Track(bugfix / 文案 / 樣式 / 設定微調 / 文件) | ✅ 跑完不必逐步確認(仍須過最小檢查) |
| **L2 Node 內自動** | 單一 Implementation Node 內 | ✅ Node 內自動跑;**Node 之間設 checkpoint** |
| **L1 草案 + 確認** | 架構決策(ADR)、資料模型定案、模組邊界 | ⚠️ 自動產出草案,**人確認後才往下** |
| **L0 必須人執行** | DB migration(真實資料)、部署上線、對外發送、刪除資料 | 🛑 一律人確認 / 人執行,**不論權限模式** |

> 判斷不確定時,往低自治走(選更需要確認的那級)。

---

## 2. 三種檢查點

| 類型 | 何時 | 行為 |
|---|---|---|
| **🛑 硬停(必停)** | L0 動作、命中 §3 清單 | 停下,等人明確確認才繼續 |
| **⚠️ 軟停(建議停)** | L1 決策、跨階段交接、跨 Node | 產出結果 + 摘要,讓人可介入;無回應可依設定續行 |
| **✅ 自動** | L2/L3 範圍內 | 直接做,結果寫入 report / log |

---

## 3. 強制人類確認的動作(硬停,不論放行模式)

這些**不可逆 / 對外**,無論權限開多大都要停:

1. DB migration 動到**真實 / 生產資料**(回填、刪欄、改型別)。
2. 部署 / 發布上線。
3. 對外發送(email、通知、webhook、第三方 API 寫入)。
4. 刪除資料 / 檔案 / 分支等不可逆操作。
5. 動到金流 / 帳號 / 權限授予。
6. 引入新的重大外部依賴 / 服務。

> 這是責任邊界,不是效率問題。release-manager 已內建此規則。

---

## 4. 強制停下回報的情境(工作流既有規則)

以下情境 AI **必須停下回報,不可自行猜測**(原始文件 §13.3):

1. **scope 不足**:Node 範圍不夠完成任務 → 回 task breakdown,不擅自擴張 Allowed Files。
2. **文件與程式衝突** → 記錄,不猜測(見 `mcp-map.md` 衝突處理)。
3. **source of truth 衝突**(Figma / 文件 / code 不一致) → 回報。
4. **需求模糊 / Open Questions 未解** → 回 product-planner。
5. **要碰 Forbidden Changes** → 停下,不突破。

> 「完全放行」不會讓這些消失。強迫 AI 跳過 = 放棄工作流的安全網。

---

## 5. 路徑 × 自治對照

| 路徑 | 自治預設 | 檢查點 |
|---|---|---|
| **Fast-Track** | L3 全自動 | 完成後過 prompt-router 的「Fast-Track 最小檢查」即可 |
| **Full-Loop** | L2 為主 | L1 決策點(04 架構 / 05 資料)軟停;L0 動作硬停;階段交接建議軟停 |

## 6. 階段檢查點地圖(Full-Loop)

| 階段 | 自治 | 檢查點 |
|---|---|---|
| 02 Planning | L2 | DoR 未過不可進開發(軟停) |
| 03 Flow | L2 | 自動;與 spec 衝突則回報 |
| 04 Architecture | **L1** | ADR 草案 → ⚠️ 人確認(會擴散) |
| 05 Data Modeling | **L1** | schema 定案 → ⚠️ 人確認;migration 真資料 → 🛑 |
| 06 UIUX | L2 | 自動;與 Figma 衝突則回報 |
| 07 Task Breakdown | L2 | Node 切分產出 → 建議軟停讓人看切得對不對 |
| 08 Implementation | L2 | Node 內自動;Node 間 checkpoint |
| 09 Test & Review | L2 | 自動;blocking 未清不可標 done |
| 10 Fix | L2 | 自動 |
| 11 Docs | L3 | 自動回寫 |
| 12 Release | **L0** | 部署 / migration → 🛑 人執行 |

---

## 7. 放行模式對應建議

把自治級別對應到實際使用:
- **想全自動跑的**:把工作限定在 Fast-Track / L3,可放心開大權限。
- **Full-Loop**:建議在 **04、05、07** 設手動確認點,其餘放行;**12 與所有 L0 動作**保持人工。
- **跨 Node 連續跑**:可自動,但建議每完成一個階段給一次摘要 checkpoint(也正好是 `context-policy.md` 的 session 切點)。

> 真正能「敢放行」的前提,是把上面這些界定清楚 —— 自動的部分才能真的全自動。

---

## 8. Subagent 與自治

- 派 subagent 執行 L2/L3 工作 → 它在自己脈絡跑完回摘要,主線審摘要即一種輕量 checkpoint。
- subagent **不得**自行執行 L0 動作;遇到要硬停的情境,回報主線,由主線升級給人。
